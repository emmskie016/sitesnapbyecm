## 1. Problem

The MVP generates every site with picsum.photos placeholder imagery (or Unsplash when keyed). Sites look like AI demos, not "this is my actual business." Customers want their own storefront photos, food shots, team headshots, logos, sometimes menus, sometimes hero videos.

## 2. Scope

### In scope (v1.1)

Three sub-features, shippable independently in three sub-phases:

- **1.1.a — Photos + Logo upload** (~1–2 days)
- **1.1.b — Document upload** (PDFs: menus, brochures) (~1 day)
- **1.1.c — Hero background video** (5–10s mp4/webm) (~2–3 days)

All uploads are **optional**. Pipeline falls back to Unsplash → picsum when a slot has no upload.

### Out of scope

- AI content moderation on uploaded media (defer; document as a known gap)
- Image super-resolution / re-touching
- Editing of uploaded media after generation
- Replacing the AI-written copy with a customer-uploaded text doc
- Per-tier upload limits (defer until Stripe ships)

## 3. Success criteria

- A customer who uploads a hero photo gets a generated site featuring **their photo**, not picsum.
- Customer who uploads nothing gets the same fallback experience as today.
- Upload validation rejects garbage (non-image files, oversized files, executables masquerading as JPEGs) before they reach R2.
- Total submission time (form submit → status=done) stays under 60s even with 10 photo uploads.
- Storage cost stays predictable: roughly 20–30 MB per submission worst case.

## 4. Architecture changes

### 4.1 API surface change — `POST /api/sites` becomes multipart

Currently the endpoint accepts JSON. v1.1 changes it to `multipart/form-data` with the following fields:

| Field | Type | Required | Notes |
|---|---|---|---|
| `full_name` | text | yes | unchanged |
| `email` | text | yes | unchanged |
| `brand_name` | text | yes | unchanged |
| `industry` | text | yes | unchanged |
| `questionnaire` | text (JSON string) | yes | client serializes the questionnaire dict to a JSON string |
| `logo` | file | no | png/jpg/svg/webp, ≤ 1 MB |
| `hero` | file | no | png/jpg/webp/heic, ≤ 5 MB |
| `gallery_1` … `gallery_10` | file | no | png/jpg/webp/heic, ≤ 5 MB each |
| `menu_pdf` | file | no | application/pdf, ≤ 5 MB (Phase 1.1.b) |
| `hero_video` | file | no | mp4/webm, ≤ 15 MB, ≤ 10 s (Phase 1.1.c) |

### 4.2 Validation pipeline

For every uploaded file, before R2 storage:

1. **Size check** — reject if over per-slot cap with HTTP 413.
2. **MIME sniff** — read first 512 bytes, validate magic bytes match declared type. Reject extensions that don't match magic.
3. **Image-specific** — `PIL.Image.open(...).verify()`; if it fails, reject as corrupt/non-image.
4. **EXIF strip** — re-save without metadata to remove GPS, device info, timestamps.
5. **Auto-orient** — apply EXIF orientation tag before stripping so portrait phone photos display right-side-up.
6. **Resize** — generate 1600w, 800w, 400w variants via Pillow for responsive `srcset`. Store all three.
7. **PDF-specific** (Phase 1.1.b) — first-bytes `%PDF-`; reject encrypted/password-protected PDFs.
8. **Video-specific** (Phase 1.1.c) — probe with `ffmpeg-python` or `pymediainfo`, reject if duration > 10s, dimensions outside reasonable bounds, codec not in `{h264, vp9, av1}`, or audio track flagged.

### 4.3 Storage layout in R2

```
uploads/<submission_id>/
  logo.png                  # original (after EXIF strip)
  hero/1600.jpg             # responsive variants
  hero/800.jpg
  hero/400.jpg
  gallery_1/1600.jpg
  gallery_1/800.jpg
  gallery_1/400.jpg
  ... up to gallery_10
  menu.pdf
  hero_video.mp4
```

Uploads are kept indefinitely (or for X days, configurable). At publish time, files are **copied within R2** from `uploads/<sub_id>/...` to `sites/<slug>/assets/...`. R2-to-R2 copies are free and instant.

### 4.4 Schema change

```sql
alter table submissions
  add column uploaded_files jsonb not null default '{}'::jsonb;
-- Shape:
-- {
--   "logo":      "uploads/<id>/logo.png",
--   "hero":      "uploads/<id>/hero",        -- prefix; variants at /1600.jpg, /800.jpg, /400.jpg
--   "gallery_1": "uploads/<id>/gallery_1",
--   "menu_pdf":  "uploads/<id>/menu.pdf",
--   "hero_video":"uploads/<id>/hero_video.mp4"
-- }
```

No `jobs` table change needed.

### 4.5 Pipeline integration

`app/pipeline/images.py` — `fetch_images_for_archetype` gains a three-tier fallback per slot:

```python
async def _fetch_one(slot, spec, ...):
    # Tier 1: customer upload
    if upload_key := submission.uploaded_files.get(slot):
        return slot, _photo_from_r2_upload(upload_key)
    # Tier 2: Unsplash
    photo = await unsplash.search_cached(...)
    if photo is not None:
        return slot, photo
    # Tier 3: picsum fallback
    return slot, _fallback_photo(archetype, slot)
```

`_photo_from_r2_upload(key)` returns an `UnsplashPhoto`-shaped object whose `url_regular` points at the **internal R2 URL** (`r2://sitesnap-sites/uploads/<sub_id>/...`). The publish stage detects this scheme and does an R2 copy operation instead of an external download.

### 4.6 Publish stage change

Currently `publish_site` downloads each image from an external URL via httpx, then uploads to R2. For customer uploads, skip the download — issue a `CopyObject` call on R2 directly:

```python
if photo.url_regular.startswith("r2://"):
    source = photo.url_regular.removeprefix("r2://sitesnap-sites/")
    await r2.copy(source=source, dest=f"sites/{slug}/assets/{slot}.jpg")
else:
    body = await _download(http, photo.url_regular)
    await r2.put(key=f"sites/{slug}/assets/{slot}.jpg", body=body, ...)
```

This makes upload-heavy submissions faster than picsum/Unsplash flows because no external bandwidth is used.

### 4.7 Template changes

- **Header / nav:** if `images.logo` is present, render `<img class="logo">` to the left of the brand name. Otherwise current text-only branding.
- **Hero:** `images.hero` is used directly if uploaded. If `images.hero_video` is also uploaded (Phase 1.1.c), the hero becomes a `<video autoplay muted loop playsinline>` with the photo as the `poster=` fallback.
- **Gallery section** (hospitality, portfolio): iterate over `images.gallery_*` keys in order. If fewer than expected, the section adapts grid size or hides.
- **Resources section** (Phase 1.1.b): if `images.menu_pdf` is set, add a new "Menu" or "Resources" section with a download link + inline first-page preview (via PDF.js client-side, or a server-rendered first-page PNG).

### 4.8 Lovable form UI changes

New section in the questionnaire, after the existing fields:

- **Logo** — single image, square crop preview, drag-drop + click-to-upload
- **Hero photo** — single image, 16:9 preview
- **Gallery photos** — multi-file picker (drag-drop), thumbnail strip with reorder + remove, max 10
- **Menu / brochure** (Phase 1.1.b) — single PDF picker
- **Hero video** (Phase 1.1.c) — single video picker with playback preview

Client-side previews only — actual upload happens on form submit (one multipart POST).

## 5. Phasing

### Phase 1.1.a — Photos + Logo (build first)

- Database migration
- Multipart endpoint
- Validation + EXIF strip + responsive resize
- Pipeline tier-1 (upload-first)
- Templates: logo in header, photos take priority over stock
- Lovable form: logo + hero + gallery slots

### Phase 1.1.b — Documents

- PDF validation (magic bytes, encryption check)
- Resources section in templates (hospitality + service archetypes)
- Lovable form: menu/PDF slot

### Phase 1.1.c — Hero video

- ffmpeg-python or pymediainfo for probe
- 10s + 15MB caps
- Hospitality template hero becomes video w/ photo fallback
- Lovable form: video slot
- *Decision deferred:* do we transcode server-side (heavy) or require customer-pre-encoded mp4?

## 6. Risks

| Risk | Mitigation |
|---|---|
| Garbage uploads burn R2 storage | Per-IP rate limit, file size caps, total submission cap (50 MB) |
| Customer uploads adult/illegal content | Defer for MVP. Add Anthropic vision moderation in a v1.2. |
| Render bandwidth bill spike on multipart uploads | Cap per-file size aggressively (5 MB photos, 15 MB video). For real volume, migrate to presigned PUT URLs (v1.2). |
| ffmpeg in Docker image bloats container | Use ffmpeg-static binary; or strip to just `ffprobe` for metadata. ~20 MB extra. |
| Photos take longer to upload than the customer's patience | Show client-side upload progress (browser-native via XHR). |

## 7. Open questions

- Should we keep `uploads/<sub_id>/` indefinitely or expire after N days? (Tradeoff: re-using a submission's photos for retries vs storage cost.)
- For Phase 1.1.c, transcode server-side or require pre-encoded?
- Per-tier upload limits — defer until Stripe ships? Yes (per scope).

## 8. Form-side UX (Lovable, ship-before-uploads)

Two UX improvements that don't require backend changes and should ship with the initial Lovable rewire (before v1.1 upload backend work):

**8.1 Multi-step questionnaire with completion percentage**

Convert the existing single long form into a multi-step wizard. Each step asks one focused topic (basics → audience → tone → goals → differentiator → optional uploads in v1.1). Top of every step shows:

- A horizontal progress bar
- Text: *"Step N of M · X% complete"*
- Back / Next buttons
- "Next" is disabled until required fields on this step are filled

Required fields per step are validated client-side; optional fields can be skipped. Completion percentage = (number of completed required fields / total required fields) × 100, recomputed on every input.

**8.2 Submission loading screen**

After the user clicks "Build My Website" but **before** redirecting to the backend's `/status/<id>` page, show an in-form loading state. Reason: there's a 1–10 second window for the multipart POST (especially once v1.1 uploads add photo+video bytes), and a blank-screen redirect feels broken.

The loading state should:

- Replace the form content in-place (don't push a modal that obscures everything)
- Show "Sending your details…" with an animated progress bar
- If the v1.1 multipart payload includes files, surface real upload progress via XHR's `progress` event ("Uploading photos: 4 of 7…")
- On 202 response, transition smoothly (200ms fade) to the redirect

The post-redirect backend status page (already implemented) takes over from there with its own progress bar for the 6-stage generation pipeline. Two screens, two different progress indicators:

1. **Lovable form** — % of questions answered (pre-submit) → upload progress (during POST)
2. **Backend `/status/<id>`** — % of generation pipeline complete (post-202)
